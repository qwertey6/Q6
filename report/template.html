PSE Detector Conformance Benchmark

Engineering report. Not a regulatory certification. See section 7 for limitations.

1. Executive summary

The benchmark runs three detectors -- our own (ours), EA IRIS (iris), and the FFmpeg vf_photosensitivity filter (ffmpeg_photosensitivity) -- against a 371-fixture corpus (306 TRACE generated videos + 8 IRIS test videos + 12 IRIS pattern images + 1 Apple sample clip + 45 fixtures we generate ourselves with analytical labels). Each detector runs once per profile per fixture; scoring matches each result against the per-fixture label for the standard that profile targets.

Headline finding: the choice of standard reading matters more than the choice of detector. Our detector under the broadcast profile (ITU-R BT.1702 or Ofcom GN2 Annex 1) scores MCC 0.273 on the upstream subset; the same detector under WCAG 2.2 SC 2.3.1 scores 0.160. The underlying code is identical -- only the threshold constants differ. EA IRIS scored against its native broadcast-flavoured threshold reading comes in at MCC 0.097; against WCAG-strict labels it scores -0.032 on the same data. That gap is not "IRIS is broken"; it is "IRIS implements broadcast thresholds and gets graded against a more conservative WCAG reading." The per-standard cross-cut below makes these apples-to-apples comparisons explicit.

2. Methodology & provenance

  • Sources: corpus/sources/ contains every upstream tree at a pinned commit; see corpus/PROVENANCE.md.
  • Standards cited: WCAG 2.2 SC 2.3.1, ITU-R BT.1702, Ofcom GN2 Annex 1, Trace24 (TRACE proposed guidelines), NAB-J. ISO 9241-391 is referenced by number only (non-free; text not vendored).
  • Labels are NOT vote-derived. Every label traces to either an upstream ground-truth file (TRACE per-set CSV, TRACE per-fixture JSON expected_result blocks, IRIS *_RELATIVE.csv) or to the analytical generation parameters (OURS-extended). See corpus/MANIFEST.csv column standard_clause, the five expected_* per-standard label columns, and generation_params.
  • Adapter label isolation: adapters take (fixture_path, profile) and never see labels. Scoring is a separate process that joins results to MANIFEST.csv after all adapter runs complete. This is the property that makes the benchmark non-gameable.
  • Reproducibility, honestly. The corpus side (make corpus) is deterministic from pinned upstream sources + pinned Python deps. The harness + scoring + report sides are similarly deterministic. Building EA IRIS in Docker is currently blocked: Debian bookworm-slim's libtool 2.4.7 is missing the LT_SYS_SYMBOL_USCORE macro that libxcrypt 4.4.36 (a transitive opencv-via-vcpkg dependency) requires. Until that's resolved, the working path is a native macOS build of IRIS via the cmake --preset macos-release preset (Homebrew's cmake, ninja, pkg-config, nasm on the host, then clone vcpkg, bootstrap, and configure). The harness detects the IRIS binary at the standard preset-output path or at /usr/local/bin/iris-example.

3. Gap analysis per competing tool

FFmpeg vf_photosensitivity

Label: mitigation, non-conformant by design. The filter has no PASS/FAIL concept and is not written to any specific PSE standard. We derive a proxy verdict: FAIL iff any flash event was detected. The filter is included as a control, showing how a non-standards approach scores; do not interpret its results as a conformance claim.

At default parameters the filter reported zero detections across the upstream FAIL fixtures and the OURS-extended FAIL fixtures -- recall 0.000 under every standard's labels. The filter's role is attenuating noisy output for sensitive viewers, not flagging hazards; the recall figure quantifies just how different that mission is.

EA IRIS (C++)

IRIS is the strongest existing standards-grounded open-source detector in the field. Built from source at the pinned 2025-01-14 commit via cmake's macos-release preset (Linux Docker build is blocked on the libxcrypt/libtool issue described in §2). The harness's IRIS adapter parses Results/<fixture>/result.json for the OverallResult + Results[] per-dimension list, treating PassWithWarning as PASS (it's informational, not a hazard verdict).

What the data says. IRIS implements broadcast / Trace24-flavoured thresholds. Under broadcast labels (ITU-R BT.1702 / Ofcom GN2) on the upstream subset, IRIS scores MCC 0.097 with FN=18 (modest but not catastrophic). Under WCAG-strict labels on the same data, MCC -0.032 with FN=82. The gap is the standards-interpretation question (§4c, OQ-4 / OQ-5), not a flaw in IRIS. On OURS-extended (where labels follow the standards text faithfully across all applicable standards), IRIS scores a perfect MCC 1.000.

Independent cross-check. The harness compares our per-frame output against IRIS's shipped *_RELATIVE.csv expected logs as a correctness anchor for the 8 IRIS test videos. One discrepancy worth flagging: GradualRedIncrease.mp4 -- our manifest derived FAIL from the shipped expected log (it has non-zero per-frame FlashRedFailedFrame entries) but IRIS's runtime classifies the same fixture as Pass. The upstream expected log appears to record per-frame events that don't aggregate to a sequence-level FAIL under IRIS's current thresholds.

Apple VideoFlashingReduction

4c. Open questions and findings

These are the standards-interpretation questions that surfaced during this work. Each is recorded with the affected fixture IDs and (where applicable) the data backing it -- never silently conformed to a label.

  • OQ-4: WCAG area threshold ambiguity -- RESOLVED via principled re-reading of the standard. WCAG 2.2 SC 2.3.1 has two defensible readings of "25% of any 10° visual field on screen (about 25% of the screen)". The strict reading operationalises the 10° field via the Harding / CRS FCS reference rectangle (341×256 px = 87,296 px ≈ 10° of central vision); 25% of that is the 21,824 px hazard threshold. The looser reading takes "about 25% of the screen" literally -- 0.25 × canvas area, which on a 1920×1080 canvas is ~518,000 px. TRACE's wcagc_* fixtures encode the strict reading (their flashing regions are around 22,000 px -- 1% of canvas, well below the loose reading but above the strict threshold). Our default WCAG2.2-SC2.3.1 profile uses the strict reading derived from the standard's own cited reference rectangle; we additionally expose a WCAG2.2-classic profile (the full 87,296 px Harding-rectangle reading) for comparison. The data in the per-standard table shows WCAG2.2-SC2.3.1 outperforms WCAG2.2-classic on TRACE labels, confirming the strict reading is the right default.
  • OQ-5: TRACE's per-fixture WCAG labels may discount the count axis -- DATA-BACKED but unresolved. Investigating fixtures like wcagc_30fps_area01/f002f038: its temporal pattern (f038_srgba.csv) cycles 222→235→248→235→222 three times over 34 frames at 30 fps -- about 1.32 flashes/sec. WCAG 2.2 SC 2.3.1's normative text requires more than three flashes within any 1-second period; 1.32 is decisively below. The per-fixture JSON nevertheless labels it wcag2_2:fail. With multi-profile scoring now in place, the data is stronger: both ours and EA iris -- both implementing count-aware thresholds -- have their lowest scores under TRACE's strict WCAG and Trace24 labels (Trace24: 0.031 for ours, -0.075 for iris). The most consistent explanation is that TRACE's strict labels apply an intensity-and-area rule that under-weights the count gate. Open question to the TRACE / WCAG-WG conversation tee'd up in the Tier 1 outreach.
  • WCAG-classic vs WCAG-strict: data confirms the strict reading is closer to TRACE's labels. ours@WCAG2.2-classic scores MCC 0.014 on upstream; ours@WCAG2.2-SC2.3.1 scores 0.160. The factor of ~10 difference comes entirely from the area threshold (21,824 px strict vs 87,296 px classic). Useful empirical input for the next round of conversation with TRACE.
  • OQ-3: Static pattern detection -- partially addressed. Our detector now ships a static bar-pattern heuristic (detect_static_pattern_hazard) that adaptively thresholds and counts bars per axis. On the 12 IRIS pattern test images it agrees with IRIS's expected labels 8/12. The four disagreements are images IRIS does NOT mark as hazardous despite visible regular striping (stripes.png, 20stripes.png, ADE.png, sacred-geometry-circle). Either IRIS applies a threshold our detector doesn't (e.g., minimum bar contrast or bar count), or IRIS's test suite is incomplete -- the IRIS pattern test data convention is "expected-result file exists ⇒ hazard", and the four images don't have matching files.
  • OQ-1: At-limit counting. WCAG says more than 3 flashes; 3 exactly is therefore PASS in our reading. A few upstream sets may treat the boundary as FAIL; recorded as disagreement rather than tuned-to.
  • OQ-2: Area at exactly 25%. Standards say more than 25%; exactly 25% is PASS in our reading. Our extended-corpus area_exactly_25pct fixture currently carries label FAIL -- this is our generator being over-strict, not the detector being wrong; left as-is rather than retuning.
  • FP-battery: equiluminant_chroma_swap. Our detector under WCAG-strict scores 2 FPs on OURS-extended, both on the FP-battery equiluminant-chroma-swap fixture (cyan↔yellow at full screen, rapid). Neither WCAG general flash (small relative- luminance Δ) nor red flash (low saturated-red on both sides) should fire, yet the per-pixel red accumulator does fire briefly at the transition. Known limitation; tracking as a refinement to the red-flash gate.
  • IRIS GradualRedIncrease.mp4 upstream data-quality flag. Already noted in §3 -- IRIS's runtime says PASS, but the shipped expected log marks per-frame red events. Not blocking our work; documented so it doesn't surprise a future reader.

7. Limitations (read this)

This report is conformance evidence against the published thresholds, gathered under independent reproducible test. It is not a regulatory certification: there is no Ofcom DPP-style authority for WCAG/web content; the broadcast standards (ITU-R BT.1702 / Ofcom GN2) name no certifying authority for third-party tooling either. For these domains this report is the strongest evidence class available.

ISO 9241-391 is referenced by number only. Its text is non-free and was not used to derive any threshold here. Where a fixture's applicability matrix says "iso", scoring against this report's results reflects the other applicable standards' thresholds.

The corpus's extended axes (frame-rate sweep, codec round-trip, color-space coverage) exercise behaviors the standards under-specify. Their results are coverage evidence, not conformance evidence.
